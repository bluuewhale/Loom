# Phase 2 — Plan 01: refinePartitionInPlace

## Goal
Eliminate per-community map allocations in `refinePartition` — the dominant Leiden-specific
allocation source identified via pprof (`alloc_objects` top: 230,989/20 = 11,549 allocs/run).

## Root Cause
`refinePartition` (called once per Leiden pass) allocated:
- `commNodes map[int][]NodeID` + inner `[]NodeID` per community
- `inComm map[NodeID]struct{}` **per community** (~1984 allocs on pass 1)
- `visited map[NodeID]bool` **per community** (~1984 allocs on pass 1)
- `commIDs []int` + `refined map[NodeID]int`

For seed 110 (4 passes, ~1984 comms pass 1): ~11,549 extra allocs/run vs Louvain.

## Solution: `refinePartitionInPlace`

Replace the old function with an in-place version that:
1. **Sorted pairs** — builds `[]commNodePair` (comm+node), sorts by (comm, node); reuses backing
2. **CSR-indexed `[]bool` scratch** — `inCommBits[csr.idToIdx[n]]` replaces per-community `inComm` map
3. **CSR-indexed `[]bool` scratch** — `visitedBits[csr.idToIdx[n]]` replaces per-community `visited` map
4. **Writes into `state.refinedPartition`** — reuses existing map (clear + repopulate)

All scratch fields added to `leidenState` struct, initialized in pool `New()`, grown lazily.

## Files Changed
- `graph/leiden_state.go`: added `commBuildPairs []commNodePair`, `inCommBits []bool`, `visitedBits []bool`
- `graph/leiden.go`: removed `refinePartition`, added `commNodePair` type + `refinePartitionInPlace`
