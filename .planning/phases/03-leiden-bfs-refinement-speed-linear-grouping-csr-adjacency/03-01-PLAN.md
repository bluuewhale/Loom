# Phase 3 — Plan 01: counting sort + CSR adjacency in refinePartitionInPlace

## Goal
Reduce ns/op overhead in `refinePartitionInPlace` after Phase 2 achieved alloc parity.
CPU profile showed `slices.partitionCmpFunc` at 3.4% flat (sort comparator), indicating
the O(N log N) sort was the main remaining Leiden-specific overhead.

## Changes

### 1. Counting sort replaces slices.SortFunc
- Pass 1: count community sizes; track dirty comm IDs via `commSeenComms`
- Sort `commSeenComms` (small list: ~comms, not ~nodes)
- Compute exclusive prefix sums → scatter offsets
- Pass 2: scatter pairs into `commSortedPairs` output buffer
- Sparse reset: zero only touched entries in `commCountScratch`
- Invariant: partition IDs always in [0, N); guarded by bounds panic

### 2. BFS queue stores int32 CSR indices (bfsQueue)
- Replaces `[]NodeID` queue → `[]int32` CSR dense indices
- `csr.adjByIdx[curIdx]` (direct slice access) replaces `g.Neighbors(cur)` (map lookup)
- Eliminates one `mapaccess2_fast64` per BFS dequeue

### 3. Hardening (from code review)
- Bounds assertion: `if comm < 0 || comm >= n { panic(...) }` — catches future phase1 regressions
- Explicit comment coupling `commSeenComms` sort order to `commSortedPairs` layout invariant
- Amortized growth for `commSortedPairs`: +25% headroom on realloc

## Files Changed
- `graph/leiden_state.go`: added `commCountScratch []int`, `commSeenComms []int`, `commSortedPairs []commNodePair`, `bfsQueue []int32`
- `graph/leiden.go`: rewrote `refinePartitionInPlace`; added `"fmt"` import
